# Simulation of Blacklist_Download (Algorithm 2)

This repository contains a full message-passing simulation of **Algorithm 2 (Blacklist_Download)** from the paper:

> *Byzantine Resilient Distributed Computing on External Data*  
> John Augustine, Anisur Rahaman Molla, Gopal Pandurangan, and Manish Kumar  
> DISC 2024

The simulation validates the paper's two central claims empirically: that the algorithm achieves **sub-linear query complexity** (Q < n) while remaining **correct for all Byzantine fractions β < 1**, even against the strongest adaptive adversary.

---

## Background

Consider a network of k peers that want to download an n-bit object from a trusted external source. Up to βk of the peers are Byzantine — they can lie, collude, and adapt their behaviour in response to everything they observe. The naive approach has each peer query the source for all n bits, giving Q = n queries per peer.

Algorithm 2 does better. Instead of every peer querying the source independently, it runs a **committee-based protocol** for each bit:

1. Each peer independently flips a biased coin and joins the committee for that bit with probability p.
2. Committee members query the source directly and broadcast their value to all peers.
3. If two committee members broadcast different values for the same bit, a conflict is detected. The source is queried to resolve it, and the peer who lied is added to a permanent blacklist.
4. Non-committee peers take a majority vote over non-blacklisted committee values and adopt the result.

The committee size parameter ρ controls the tradeoff between the primary cost (np, paid when peers join the committee) and the verification cost (βk/ρ, paid when conflicts trigger source queries). Setting:

```
ρ* = max(1, k · sqrt(β·γ / (8n)))    where γ = 1 - β
p  = min(1, (6 ln n + 4ρ) / (γ·k))
```

minimises total query complexity. The paper proves Q = O(n log n / (γk) + sqrt(n)) w.h.p., which is sub-linear in n when k is large enough.

## Implementation

Every agent in the simulation is a real HTTP server. The source is a separate HTTP server. Every committee broadcast and every source query is an actual network request. There are no shared memory shortcuts. The simulation has been tested with up to k = 700 agents on a single laptop.

### Key flags

| Flag | What it controls | Default |
|---|---|---|
| `-agents` | k, number of peers in the network | 100 |
| `-bits` | n, number of bits in the object | 256 |
| `-ratios` | Byzantine fraction β; comma-separated, fractions like `1/3` work | `1/3` |
| `-adversary` | `random`, `collude`, `flood`, or `drip` | `flood` |
| `-rho` | committee size ρ; 0 derives the paper's optimal value | 0 (auto) |
| `-rho-divisor` | divisor inside ρ's formula (paper uses 8) | 8 |
| `-p-ln-coeff` | ln(n) coefficient in p (paper uses 6) | 6 |
| `-p-rho-coeff` | ρ coefficient in p (paper uses 4) | 4 |
| `-seed` | random seed for reproducibility | 42 |
| `-csv` | path to append result rows (CSV format) | — |
| `-verbose` | per-round trace: committee elections, vote tallies, conflicts | false |
| `-demo` | animated colour terminal visualisation (good for small k, n) | false |

The `drip` adversary exposes exactly ρ+1 Byzantine peers each round to force a conflict, representing the strongest possible adaptive strategy. All experiments in this study use drip.

## Experiments and results

Three experiments were run to validate the algorithm's behaviour. All results are in `benchmarks/results/`.

### Experiment 1 — Query complexity vs problem size

**Question:** As n grows, does the measured Q track the theoretical prediction Q = np?

**Setup:** k = 100 peers, n = 512 / 1024 / 2048 bits, β ∈ {0, 1/3, 1/2, 2/3}, drip adversary, 30 independent seeds per (n, β) configuration (360 total runs on HPC).

**Results:**

| n | β | Q measured (mean ± std) | Q theory (np) | Naive Q = n |
|---|---|---|---|---|
| 512 | 0 | 216 ± 0 | 212 | 512 |
| 512 | 1/3 | 341 ± 1 | 318 | 512 |
| 512 | 1/2 | 456 ± 1 | 424 | 512 |
| 512 | 2/3 | 545 ± 0 | — | 512 |
| 1024 | 0 | 476 ± 0 | 467 | 1024 |
| 1024 | 1/3 | 724 ± 2 | 700 | 1024 |
| 1024 | 1/2 | 967 ± 1 | 934 | 1024 |
| 1024 | 2/3 | 1057 ± 0 | — | 1024 |
| 2048 | 0 | 1027 ± 0 | 1019 | 2048 |
| 2048 | 1/3 | 1545 ± 0 | 1528 | 2048 |
| 2048 | 1/2 | 2070 ± 2 | 2038 | 2048 |
| 2048 | 2/3 | 2081 ± 0 | — | 2048 |

For β ∈ {0, 1/3, 1/2}, measured Q tracks the theory curve within 8% at n = 512 and tightens to under 2% at n = 2048. All curves stay well below the naive Q = n baseline.

At β = 2/3, the committee probability p saturates to 1 (every peer joins every committee for every bit), so the primary cost equals n. The measured Q slightly exceeds n because the verify cost (source queries during conflict resolution, βk/ρ) adds on top. The theory column is marked "—" for these rows because the figure plots only the primary component np; the total cost including verify queries is above n by design at this extreme Byzantine fraction.

**Figure:** `benchmarks/graphs/fig1_Q_vs_n.png` — log-log plot of Q vs n for all four β values; markers are measured means with error bars (±1 std), solid lines are the theoretical np prediction, dashed line is the naive Q = n baseline.

### Experiment 2 — Correctness vs Byzantine fraction

**Question:** Does the protocol output the correct answer for every β < 1, even under the strongest adversary?

**Setup:** k = 120, n = 64, β swept from 0.05 to 0.90 in 10 steps, drip adversary, 30 independent seeds per β value (300 runs total). Extended runs at n = 128, 256, 512 with 30 seeds each (900 additional runs).

**Results (n = 64):**

| β | Seeds | Successes | Success rate | Wilson 95% lower bound |
|---|---|---|---|---|
| 0.05 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.10 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.20 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.30 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.40 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.50 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.60 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.70 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.80 | 30 | 30/30 | 100% | ≥ 88.6% |
| 0.90 | 30 | 30/30 | 100% | ≥ 88.6% |

Zero failures across all 300 runs, including at 90% Byzantine. Results at larger n are identical. The Wilson 95% confidence interval is reported because a raw "30/30 = 100%" overstates certainty; the lower bound of ≥ 88.6% is the honest claim from finite trials.

### Experiment 3 — Sensitivity of the paper's constants

**Question:** The paper uses specific constants (6 and 4) in the committee probability formula. How much slack is built into them?

**Setup:** k = 120, n = 64, β ∈ {0.1, 0.3, 0.5, 0.7, 0.9}, five constant configurations ranging from the paper's values down to 1/16 of them. 50 seeds per configuration (1250 total runs). A separate β = 0.5 run used 50 seeds (250 runs).

The five configurations tested, where committee probability is `p = min(1, (a·ln n + b·ρ) / (γk))`:

| Config | a | b | Honest committee size (μ) at β=0.5 |
|---|---|---|---|
| Paper | 6 | 4 | 35.6 |
| Halve ×1 | 3 | 2 | 17.8 |
| Halve ×2 | 1.5 | 1 | 8.9 |
| Halve ×3 | 0.75 | 0.5 | 4.5 |
| Halve ×4 | 0.375 | 0.25 | 2.2 |

**Results (success rate across all β values):**

| Config | β=0.1 | β=0.3 | β=0.5 | β=0.7 | β=0.9 |
|---|---|---|---|---|---|
| Paper | 100% | 100% | 100% | 100% | 100% |
| Halve ×1 | 100% | 100% | 100% | 100% | 100% |
| Halve ×2 | 100% | 100% | 100% | 100% | 100% |
| Halve ×3 | 100% | 100% | 2% | 0% | 44% |
| Halve ×4 | 0% | 0% | 0% | 0% | 0% |

The protocol is fully correct down to one quarter of the paper's constants. The failure at Halve×3 coincides with the honest committee losing its majority — when μ drops to around 4.5 honest members, Byzantines can occasionally outvote them. The anomalous 44% at (Halve×3, β=0.9) is a genuine result: at very high Byzantine fractions the drip adversary's budget shrinks (fewer unblacklisted Byzantines to expose), which paradoxically makes conflicts less frequent and slightly improves the outcome for reduced constants.

**Figure:** `benchmarks/graphs/fig2_constants_heatmap.png` — 5×5 heatmap of success rate by config and β, green for 100%, red for failure. The sharp cliff between Halve×2 and Halve×3 is clearly visible.

## Notes on the adversary model

The drip adversary is the strongest setting tested here. Each round it selects exactly ρ+1 Byzantine peers who have not yet been blacklisted and has them produce conflicting votes, forcing a source query for conflict resolution. Once blacklisted, a Byzantine peer is silent for the remainder of the run. As the blacklist grows, the drip adversary runs out of fresh Byzantines to expose and the conflict rate drops — this is precisely the mechanism the paper relies on for its query complexity bound.

The flood adversary (all Byzantines lie simultaneously) is used in the canonical sanity run; it is stronger in terms of the lie broadcast but not in terms of forcing source queries, because blacklisting happens faster.
